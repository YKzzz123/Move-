# Move! V2 项目开发流程

## 文档约定
- 本文档用于记录项目开发过程中的模块新增、功能更新与关键决策。
- 规则：每次新增模块或功能后，必须同步更新本文档。
- 记录格式建议：时间、变更类型、影响范围、核心内容、后续待办。

## 开发记录

### 2026-04-26

#### 1) 初始化开发环境
- **变更类型**：环境初始化
- **影响范围**：项目根目录
- **核心内容**：
  - 创建虚拟环境：`moveV2`
  - 安装基础后端依赖：`fastapi`、`uvicorn`、`sqlalchemy`、`pymysql`、`python-dotenv`、`google-generativeai`、`pydantic-settings`、`alembic`
  - 生成依赖锁定文件：`requirements.txt`

#### 2) 数据层初始化
- **变更类型**：新模块
- **影响范围**：`move-backend/database.py`、`move-backend/models.py`
- **核心内容**：
  - 新建数据库连接模块 `database.py`
    - 基于 SQLAlchemy + PyMySQL 配置 MySQL 同步连接池
    - 支持环境变量 `MOVEV2_DATABASE_URL`（默认连接本地 MySQL）
    - 提供 `SessionLocal`、`get_db()`、`create_tables()`、`generate_create_table_sql()`
  - 新建 ORM 模型模块 `models.py`
    - 定义 `User`、`SessionRecord`（sessions）、`Diary`、`EnergyStation` 四张表
    - 添加外键关系、类型提示、字段长度限制与时间戳字段

#### 3) API 模块新增（气值与疗愈文案）
- **变更类型**：新模块 + 新功能
- **影响范围**：`move-backend/main.py`
- **核心内容**：
  - 新增接口 `POST /api/users/{user_id}/qi`
    - 接收 `qi_delta` 并更新 `User.qi_score`
    - 包含用户不存在与数据库异常处理（事务回滚）
  - 新增接口 `POST /api/workouts/complete`
    - 接收“运动完成”信号
    - 调用 Gemini 生成疗愈文案
    - 将文案写入 `EnergyStation`
    - 包含 API Key 缺失、模型调用失败、数据库异常处理

#### 4) AI 服务层抽离与能量金句接口
- **变更类型**：新模块 + 新功能
- **影响范围**：`move-backend/services/__init__.py`、`move-backend/services/gemini_service.py`、`move-backend/main.py`
- **核心内容**：
  - 新建 `services/gemini_service.py`
    - 使用 `python-dotenv` 从根目录 `.env` 读取 `GEMINI_API_KEY`
    - 提供异步函数 `generate_healing_quote(performance) -> (quote, is_default)`
    - 生成结合东方古典 / 中医意境的鼓励金句，长度限制 30 字以内
    - 自动 sanitize（去 Markdown、引号、换行），并支持长度截断
    - API 调用失败 / Key 缺失 / 空响应时，返回优雅默认文案 `气定神闲，一念一春。`
  - 新增接口 `POST /api/energy/generate`
    - 入参：`user_id`、`performance`（运动表现描述）
    - 校验用户存在 → 调用 service 生成金句 → 写入 `EnergyStation`
    - 通过 `source_tag` 区分 `gemini` 与 `gemini-default`
    - 数据库异常自动回滚，返回 500
- **后续待办**：
  - `.env` 中需要补充 `GEMINI_API_KEY=...`
  - 后续可考虑将 `complete_workout` 接口也统一改为调用 `gemini_service`，避免重复逻辑

#### 5) AI 双通道容灾（Gemini -> 豆包）
- **变更类型**：功能增强
- **影响范围**：`move-backend/services/gemini_service.py`
- **核心内容**：
  - 新增“优先 Gemini，失败自动切换豆包 API”的容灾策略
  - 新增豆包配置项读取：
    - `DOUBAO_API_KEY`
    - `DOUBAO_BASE_URL`（默认 `https://ark.cn-beijing.volces.com/api/v3/chat/completions`）
    - `DOUBAO_MODEL`（默认 `doubao-1-5-lite-32k-250115`）
  - 调用链路调整为：
    1. Gemini 成功返回 -> 直接使用  
    2. Gemini 失败/无 Key/空内容 -> 调用豆包  
    3. 豆包也失败 -> 返回默认文案 `气定神闲，一念一春。`
  - 继续保持 30 字限制与文本 sanitize 逻辑
- **后续待办**：
  - 在 `.env` 中补充豆包配置，确保备用链路可用

- **核心问题**：旧库 `users` 表无 `username` / `hashed_password` 时，注册 INSERT 会失败。
- **后端**：
  - `database.ensure_user_auth_columns()`：应用启动时（`main.on_startup` 在 `create_tables` 之后）对 **MySQL** 检测 `information_schema`，缺列则 `ALTER TABLE` 补 `username`、`hashed_password`，并尝试建唯一索引 `uq_users_username`。
  - 手工 SQL 备份：`move-backend/scripts/add_users_auth_columns.sql`
  - `POST /api/auth/login`：`LoginRequest` + `bcrypt` 验密，返回与 `UserResponse` 相同字段。
- **前端**：`src/views/Login.vue`、`/login` 路由、顶栏「登录」、注册页链至登录；`userStore.login()`。
- **联调验证**：`npm run build`、后端 `py_compile` 通过。**需重启 uvicorn** 以执行启动迁移。

#### 21) 注册 500：passlib 与 bcrypt 5.x 不兼容修复
- **问题现象**：前端仅显示「注册未成功」；直连 `POST /api/auth/register` 返回 500。
- **根因**：`passlib==1.7.4` 依赖 `bcrypt.__about__.__version__`，与 `bcrypt 5.x` 不兼容，`hash_password` 在首次哈希时抛错。
- **修复**：`auth/utils.py` 改为 **直接使用 `bcrypt` 库** 做 `hashpw` / `checkpw`，从 `requirements.txt` 移除 `passlib`。
- **旁路**：`main.py` 注册失败时 `detail` 附带数据库原始错误信息，便于排查缺列等问题；`Register.vue` 对无 `response` 时提示检查 uvicorn 与（开发）Vite 代理是否生效。
- **注意**：修改后需 **重启 uvicorn** 以加载新 `auth.utils`。

#### 22) Vite 端口、代理与 CORS
- **端口**：`port: 5173`，`strictPort: false`，被占用时自动递增到 5174、5175…
- **开发代理（避免 Network Error）**：页面在 `127.0.0.1:517x` 时，若 Axios 直连 `localhost:8001`，Windows 上 `localhost` 常解析为 **IPv6**，后端只监听 **127.0.0.1:8001** 会 **ECONNREFUSED**。已配置 `server.proxy['/api'] -> http://127.0.0.1:8001`，且 `request.js` 在 **DEV** 下 `baseURL` 为空，请求走同域 `/api/...` 由 Vite 转发；开发 **不必** 在 `.env` 写 `VITE_API_BASE_URL`。
- **后端** `CORSMiddleware`：`allow_origin_regex` 匹配本机任意前端端口；直连模式仍可用。
- **注意**：修改 CORS / 代理后需 **重启 uvicorn** 与 **npm run dev**。
- **勿多开后端**：不要同时起 **多个** uvicorn 占 `127.0.0.1:8001`，否则可能连到旧进程、出现难复现的 500。用 `netstat -ano | findstr :8001` 看 PID，结束多余 **python** 后再只启动一个。
- **注册与联调**：`main.py` 对 `IntegrityError`（并发重名）返回 400；`GET /api/health` 用于检查数据库；API 地址联调为 `http://127.0.0.1:8001`（勿与前端端口 517x 混淆）。开发可用 `move-backend/start_dev.ps1` 或 `move-backend/scripts/run_dev_backend.ps1` 先**清掉 8001 再启动**；**勿**在 `move-backend\scripts` 下直接运行 `uvicorn main:app`（会报无法导入 `main`，因该目录无 `main.py`）。
- **仍见 `{"detail":"Not Found"}`**：多为 **8001 上不是本项目**或 uvicorn 已停。用 `netstat -ano | findstr :8001` 查 PID，在 **`move-backend` 目录**重新启动后端并保持终端不关。
